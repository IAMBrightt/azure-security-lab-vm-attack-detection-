1. Creating the Azure Environment
First, I created a free Azure subscription and logged into the Azure Portal.
After accessing the portal, I built the basic infrastructure needed for the lab.
Steps performed:
•	Created a Resource Group
•	Created a Virtual Network (VNet)
•	Deployed a Windows Virtual Machine
•	Attached the VM to the virtual network
•	Assigned a public IP address
Once the VM was deployed, the next step was configuring the network to allow internet traffic to reach the machine.
 
2. Configuring the Honeypot
To simulate a vulnerable system, I configured the Network Security Group (NSG) associated with the virtual machine.
I created a rule that allowed all inbound traffic. This intentionally exposes the system to the internet so that it can attract login attempts from external sources.
Next, I connected to the virtual machine.
Since I am using macOS, I used the Microsoft Remote Desktop application to connect to the Windows VM.
Once logged into the system, I disabled the Windows Firewall to ensure that external traffic would not be blocked by the operating system.

At this point, the virtual machine was acting as a honeypot exposed to the public internet.
![image](https://github.com/IAMBrightt/azure-security-lab-vm-attack-detection-/blob/0bc3b35a9924f00e0d776562218c9fdc49228dfa/Inbound%20Rules%20Created2.png)

4. Testing Connectivity
To confirm that the virtual machine was reachable from the internet, I attempted to ping the public IP address of the VM from my local machine.
If the ping request succeeds, it confirms that the machine is reachable from outside the Azure environment. This means that attackers could also potentially reach the machine.
This step verifies that the honeypot is successfully exposed to the internet.
![image](https://github.com/IAMBrightt/azure-security-lab-vm-attack-detection-/blob/0bc3b35a9924f00e0d776562218c9fdc49228dfa/Pinged%20IP.png)

6.  Generating Failed Login Attempts
Next, I generated some test activity on the virtual machine.
I attempted several failed login attempts using a fake username such as:
employee
These failed login attempts generate Windows Security Events.
Inside the virtual machine, I opened Event Viewer and navigated to:
Windows Logs → Security
Here I was able to observe Event ID 4625, which represents failed login attempts.
These logs include information such as:
•	Username used
•	Source IP address
•	Timestamp
•	Failure reason
![image](https://github.com/IAMBrightt/azure-security-lab-vm-attack-detection-/blob/0bc3b35a9924f00e0d776562218c9fdc49228dfa/Windows%20Event%20Log.png)

7.  Creating a Central Log Repository
To analyze these logs more effectively, I created a Log Analytics Workspace in Azure.
The Log Analytics Workspace acts as a central repository for collected logs.
Once created, I configured the virtual machine to send its security logs to this workspace.
This allows us to query and analyze security events from outside the VM.

8. Integrating Microsoft Sentinel (SIEM)
Next, I deployed Microsoft Sentinel, which functions as a Security Information and Event Management (SIEM) platform.
Sentinel was connected to the Log Analytics Workspace so that it could access the logs generated by the virtual machine.
After that, I configured the Windows Security Events via AMA connector, which allows Windows security logs to be forwarded into the workspace.
This step enabled Sentinel to begin collecting authentication events and other security activity from the honeypot.

9.  Querying Logs with KQL
Once logs were flowing into Log Analytics, I used Kusto Query Language (KQL) to search through the collected events.
Example query to view failed login attempts:
SecurityEvent
| where EventID == 4625
| order by TimeGenerated desc
This query retrieves all failed authentication events recorded on the system.
Understanding how to query logs using languages like KQL, SQL, or SPL is an essential skill for working in a SOC environment.

11. Enriching Logs with Geographic Data
The security logs contain attacker IP addresses, but they do not include geographic information by default.
To enrich the data, I imported a GeoIP dataset into Microsoft Sentinel as a Watchlist.


The watchlist contains thousands of IP address ranges mapped to geographic locations.
After importing the dataset, it becomes possible to match attacker IP addresses to their approximate geographic origin.

9.  Mapping Attacker Locations
Using the GeoIP watchlist, I ran the following query to enrich the logs:
let GeoIPDB_FULL = _GetWatchlist("geoip");
let WindowsEvents = SecurityEvent
| where EventID == 4625
| order by TimeGenerated desc
| evaluate ipv4_lookup(GeoIPDB_FULL, IpAddress, network);
WindowsEvents
This query links IP addresses found in failed login attempts with geographic location data.
As a result, we can see where the login attempts are originating from.

10.  Creating the Attack Map
To visualize attacker activity, I created a Microsoft Sentinel Workbook.
Inside the workbook, I added a query visualization that displays attack sources on a world map.
This attack map shows the geographic locations where login attempts originate.
Visualizing attacks in this way can help analysts quickly identify patterns or suspicious activity coming from certain regions.
